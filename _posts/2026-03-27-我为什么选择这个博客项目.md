一开始想做自己的博客时，我其实有点犹豫。

如果完全从零开始搭，我会很容易卡在各种细节里：服务器怎么搞、域名怎么配、页面怎么写、文章怎么发布、以后怎么维护。对我这种刚开始折腾个人博客的人来说，这些问题加在一起，门槛确实有点高。

所以后来我选了现在这个博客项目。原因很简单：它已经能跑起来，我可以先把博客用起来，再慢慢理解它背后到底用了什么技术、这些技术分别在做什么。对我来说，这种方式比一上来就从零造轮子更友好。

## 1. 这个博客背后用了哪些技术

我现在是这样理解这套博客项目的：

- `GitHub Pages` 负责把网站放到网上，让别人能访问
- `Jekyll` 负责把我写的文章和模板拼起来，生成最终页面
- `Markdown` 负责让我专心写内容，而不是一直写 HTML
- `Bootstrap` 负责页面的基础布局和样式
- `Less` 负责管理样式代码
- `Grunt` 负责处理一部分前端资源
- `kramdown` 和 `Rouge` 负责 Markdown 解析和代码高亮
- `jekyll-paginate` 负责首页文章分页
- `service worker` 和 `PWA` 相关文件负责缓存和离线能力

如果只是第一次接触这些名字，确实会觉得有点多。但我后来发现，不需要一开始把它们全都学会，只要先知道每个东西大概是干什么的，就已经足够往前走了。

## 2. 这些技术在项目里分别负责什么

我自己现在会把它理解成一条很简单的链路：

我先写文章，然后 Jekyll 读取文章，再把文章套进模板里，最后 GitHub Pages 帮我发布出去。

具体一点说就是：

1. 我在 `_posts` 目录里写 Markdown 文章
2. Jekyll 读取这些文章
3. Jekyll 再读取 `_layouts` 和 `_includes` 里的模板
4. 它把文章内容和模板拼接成完整网页
5. 我把改动提交并推送到 GitHub
6. GitHub Pages 自动重新构建网站

对我来说，这里面最重要的转变是：我不再把“发博客”理解成“手写一个网页”，而是理解成“把一篇文章交给网站生成器去处理”。

这样一来，我要关注的重点就从“网页代码怎么写”变成了“文章内容怎么写”。这个感觉轻松很多。

## 3. 我是怎么理解这个项目目录结构的

刚看到这个仓库的时候，我其实也有点懵。后来我会把它看成一张地图：

- `_posts`：文章内容放这里
- `_layouts`：页面的大骨架放这里
- `_includes`：可以重复复用的小零件放这里
- `css`、`js`、`img`：网站前端资源放这里
- `less`：样式的源码放这里
- `_config.yml`：全站配置基本都在这里

如果用最直白的话来说：

- 我写什么，主要看 `_posts`
- 页面长什么样，主要看 `_layouts`
- 网站整体设置是什么，主要看 `_config.yml`

这样理解以后，整个项目就没那么吓人了。它不再像一个“完全看不懂的模板仓库”，而更像一个分工很清楚的博客项目。

## 4. 我踩到的几个坑

这个项目我在上手时，确实踩了几个很典型的坑。

第一个坑，是我以为文章 push 上去就一定会马上出现在首页。后来才发现，如果文章日期写成“未来时间”，GitHub Pages 在构建的时候可能会把它当成 future post，于是首页就什么都没有。

第二个坑，是我明明改了内容，网页上却像没变化一样。这个项目启用了 `service worker`，浏览器有缓存，所以有时候不是网站没更新，而是我看到的是旧缓存。后来我就学会了用 `Ctrl + F5` 强刷，或者直接开无痕窗口检查。

第三个坑，是我刚开始会把“本地保存文件”和“线上更新博客”混在一起。实际上这两件事根本不是同一回事。真正让博客更新的是：

```bash
git add .
git commit -m "update post"
git push origin master
```

只有代码被推到 GitHub，GitHub Pages 才会重新构建。

这些坑现在回头看并不复杂，但如果没人提醒，新手真的很容易在这些地方卡住。

## 5. 为什么我觉得这套方案适合新手

我现在越来越觉得，这套方案很适合像我这种刚开始做个人博客的人。

原因不是它“最先进”，而是它“够用而且清晰”。

它不用我自己买服务器，不用我自己搭数据库，也不用我先学会一整套复杂的后端开发流程。我只需要知道：

- 文章放在哪
- 配置改哪里
- 改完以后怎么 push

这就已经足够让我开始写博客了。

而且它还有一个我很喜欢的地方：如果我以后想继续深入，也完全有空间。

我可以慢慢去学模板结构，慢慢去学 CSS，慢慢去学 Jekyll 的变量和页面生成方式。也就是说，它既允许我先上手，也允许我以后继续折腾。

## 6. 我接下来准备怎么继续把它变成自己的博客

对我来说，这个博客现在还只是一个起点。

我已经把旧项目残留的文章和页面清掉了，也开始写属于自己的文章了。接下来我还想继续做几件事：

- 补一个真正属于我自己的 About 页面
- 再写几篇和博客搭建有关的记录
- 慢慢把首页、标签和内容整理得更清楚
- 之后再考虑要不要继续美化样式和导航

我现在的想法不是追求“一次性全做完”，而是先让这个博客持续长起来。

## 7. 写在最后

如果现在再回头看，我会觉得这个博客项目最适合我的一点，不是它有多复杂，而是它给了我一个很好的起点。

它让我不需要一开始就懂完所有技术，也能先把自己的博客搭起来；同时，在我愿意继续学的时候，它又不会把我限制住。

对我来说，这就已经很够了。

先写起来，先发出来，先让博客真正属于自己。剩下的，再慢慢学。
