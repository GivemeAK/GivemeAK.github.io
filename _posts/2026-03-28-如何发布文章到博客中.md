如果你已经 fork 了这个博客项目，那么以后发布新文章其实很简单。  
整个流程可以记成一句话：

> 在 `_posts` 里写 Markdown，提交到 GitHub，等待 GitHub Pages 自动构建。

下面我把它拆成最实用的几个步骤，方便以后直接照着做。

## 1. 新建文章文件

文章都放在 `_posts` 目录下。文件名建议使用下面这种格式：

```text
YYYY-MM-DD-文章标题.md
```

例如：

```text
2026-03-28-如何发布文章到博客中.md
```

这个格式很重要，因为 Jekyll 会根据文件名判断文章日期，并自动生成文章链接。

## 2. 写好文章头部信息

每篇文章开头都要写一段 front matter，也就是三个短横线包起来的元数据。

```yaml
---
layout: post
title: 如何发布文章到博客中
subtitle: 从写作到上线，只需要几个固定步骤
date: 2026-03-28
author: GivemeAK
header-img: img/post-bg-desk.jpg
catalog: true
tags:
    - 博客搭建
    - GitHub Pages
    - Jekyll
---
```

这些字段分别表示：

- `layout`：文章使用的布局模板
- `title`：文章标题
- `subtitle`：文章副标题
- `date`：文章发布时间
- `author`：作者名
- `header-img`：文章顶部头图
- `catalog`：是否显示目录
- `tags`：文章标签

如果你只是想先发文章，不想折腾太多样式，先把这些字段照抄就够了。

## 3. 开始写正文

正文就是普通的 Markdown 内容，你可以直接写标题、列表、引用、代码块。

一个常见的发布流程可以写成这样：

1. 在 `_posts` 目录新建一个 Markdown 文件
2. 写好 front matter
3. 填充正文内容
4. 本地检查文章效果
5. `git add`、`git commit`、`git push`
6. 等待 GitHub Pages 自动更新

如果你在正文里想放操作命令，也可以直接写：

```bash
git add .
git commit -m "Add new blog post"
git push origin master
```

## 4. 提交到 GitHub

写完文章后，最关键的一步是把改动提交到仓库。

```bash
git add _posts/2026-03-28-如何发布文章到博客中.md
git commit -m "Add post: how to publish a blog article"
git push origin master
```

只要你的仓库已经接到了 GitHub Pages，push 之后 GitHub 就会自动重新构建网站。

## 5. 等待页面更新

一般来说，推送完成后 GitHub Pages 会自动更新。

如果你发现网页没有立刻变化，可以先检查这几件事：

- 文章文件是否真的已经 push 到 GitHub
- 文件名是否符合 `YYYY-MM-DD-标题.md`
- `date` 是否写成了正确的发布日期
- 浏览器是否缓存了旧页面
- 是否需要强制刷新页面

这个项目还启用了 Service Worker，所以有时浏览器会缓存旧资源。  
如果你明明已经更新了文章，但页面看起来没变，可以试试：

- `Ctrl + F5` 强制刷新
- 用无痕窗口打开
- 清理站点缓存

## 6. 你真正需要记住的事情

如果只记一句话，那就是：

- **本地改文章不会自动上线**
- **必须提交并推送到 GitHub**
- **GitHub Pages 才会重新生成页面**

所以完整流程其实就是：

```text
写文章 -> 保存到 _posts -> 提交 -> 推送 -> 等待 GitHub Pages 更新
```

## 7. 小结

以后你往这个博客里发文章，只要按下面的方式做就行：

- 新建 `_posts/YYYY-MM-DD-标题.md`
- 写好 front matter
- 用 Markdown 写正文
- `git commit`
- `git push`
- 等 GitHub Pages 自动刷新
